
**Coupling Map**
- Core session orchestration (TorrentManager + SessionService + Core) owns the `libtorrent::session`, queues, RPC ID maps, torrent handles, start/stop/add/remove actions, and direct calls such as `h.force_recheck()`/`force_reannounce()`/`resume()`/`pause()`, so any adapter must supply a session abstraction, hash-id translation, and the same lifecycle hooks while converting alerts into the existing callback/event model (see TorrentManager, SessionService, Core citations). - Snapshot & event telemetry (SnapshotBuilder, Events, AlertRouter, TorrentUtils) dig into handles for files, trackers, peers, and alert payloads and publish rich DTOs and events, so an adapter must provide equivalents of `peer_info`, piece states, alert-to-event translations, and SHA1 helpers to keep the snapshot bus intact. - Persistence, resume, and configuration (ResumeDataService + SettingsManager + BlocklistManager) consume `libtorrent::add_torrent_params`, `write_resume_data`, `settings_pack`, and `ip_filter`, so an adapter must expose resume serialization hooks, settings translation, and IP-filter rule application that mirror the current data flow. - Automation and completion hooks (AutomationAgent) rely on `libtorrent::torrent_handle/`status for seed ratio logic, so a replacement engine must publish the same completion callbacks and status flags to keep automated ratio/idling behaviors stable. AutomationAgent.hpp AutomationAgent.cpp

**Binary Size Forensics**
- Release/minsize builds inject `/Os`, `/GL`, `/MT`, and `/LTCG` into both C and C++ compile/link flags, so the compiler already uses static CRT and LTO-level optimizations when shipping the 12 MB binary. meson.build
- The dependency bundle links against `LibtorrentRasterbar` (via its CMake target) and statically pulls in `yyjson`, `unofficial::sqlite3`, plus the vendor `libtorrent/partfile_extension` and `tinytorrent` packed resources, so third-party runtimes and serialization helpers are already compiled-in. meson.build
- Core system libraries (`Ws2_32`, `Advapi32`, `Winhttp`) are explicitly linked. The dominant size likely owes to the static runtime and `libtorrent` machinery, so the net gain from rewriting the daemon frontend in C cannot be asserted from this configuration alone (unknown impact beyond these link flags). meson.build

**Questions**
1. How invasive is replacing libtorrent (what % of modules touch lt types)?  
   The `tt_base_sources` list enumerates ~26 engine files, and at least the majority reference `libtorrent` (TorrentManager, SessionService, Core, SnapshotBuilder, BlocklistManager, ResumeDataService, AutomationAgent, AlertRouter, SettingsManager, TorrentUtils, Events plus their headers). Each of those files includes/wraps `libtorrent` handles, alerts, or settings, so replacing libtorrent would touch most of the engine’s source units (see the source list and the cited files). meson.build TorrentManager.hpp SessionService.cpp Core.cpp SnapshotBuilder.cpp BlocklistManager.cpp ResumeDataService.cpp AutomationAgent.cpp AlertRouter.cpp SettingsManager.cpp TorrentUtils.hpp
2. Can you isolate via an adapter without rewriting the whole daemon?  
   The RPC dispatcher and helpers do not include `libtorrent` headers at all; they operate on engine DTOs and JSON utilities, so the boundary is cleanly between the RPC layer and the engine (`Dispatcher` includes only RPC/utility headers). Wrapping the engine side (SessionService/TorrentManager) with an adapter allows RPC to continue consuming snapshots/events exactly as today. Dispatcher.cpp SessionService.hpp
3. Does rewriting in C reduce size based on what the build currently does?  
   Release builds already compile with `/Os`/`/GL`/`/MT` and `/LTCG`, link against static dependencies (`yyjson`, `unofficial::sqlite3`, `LibtorrentRasterbar`, plus vendor sources), and pull in Windows system libraries. This configuration already minimizes size, so there is no direct evidence that rewriting to C would meaningfully shrink the binary beyond the current C++ + static dependency footprint. meson.build meson.build
4. Minimum experiment to validate feasibility?  
   Implement a stub `TorrentManager`/adapter providing the same API (session start, `enqueue_task`, `torrent_handles`, `set_ip_filter`, `async_add_torrent`, alert callbacks, snapshot builders) and wire it behind `SessionService`/`SnapshotBuilder`. If the stub can satisfy `SessionService` calls and produce the same snapshots/events (see the APIs in these headers), it proves the rest of the daemon can consume a non-libtorrent engine. TorrentManager.hpp SessionService.hpp SnapshotBuilder.hpp AlertRouter.hpp
